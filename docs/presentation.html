the problem

Sizing analog circuits is a hand-run loop

Every transistor has a width, length, and finger count. Picking them to hit speed, power, offset, noise targets means editing a netlist, running SPICE, reading the numbers, and adjusting — over and over, then re-checking across process corners.

the ask

Close the loop

Let an optimizer drive the simulate→adjust cycle instead of a human.

the ask

Sign it off

PVT corners, mismatch yield, layout + DRC — not just a nominal point.

the ask

See it

A real console: schematics, waveforms, tuning curves, Pareto fronts.

architecture

One SPICE backend, two front doors

Dependency-free Python stdlib bridge — no framework. ngspice runs as a subprocess (releases the GIL), so every optimizer and corner sweep is parallelized across cores.

the connection

Same regenerative core, two jobs

The comparator's cross-coupled latch and the LC-VCO's −gm pair are the identical positive-feedback structure.

  • The comparator uses it to decide once — regenerate to a rail.
  • The VCO uses the same −gm to oscillate forever — cancel the tank loss.

optimization algorithms

Differential Evolution, made cheap

global search

Log-space DE

Searches device widths in log space, minimizes power under penalty constraints — beats greedy descent.

fewer sims

GP surrogate

A Gaussian-process model on log(cost) pre-screens clearly-worse candidates → ~35–45% fewer SPICE runs.

trade-offs

NSGA-II Pareto

deep dive · vco phase noise

Three views — and an honest wall

analytic

Thermal L(Δf)

√(kT·C)/I per transition → 1/f² region + FoM; a flicker 1/f³ term at an assumed corner.

measured

SPICE trnoise

Inject physical device noise, measure period jitter over 4 seeds → agrees within ~1 dB.

measured

Jitter slope

σΔt(τ) accumulation → 0.5 = thermal, 1.0 = flicker. Confirms thermal-limited.

honest scope  A brute-force transient resolves offsets only down to ≈ f₀/Ncycles — the flicker corner is genuinely out of reach and needs a PSS/pnoise engine. The tool says so.

engineering · performance

Measured, then made fast

48×sky130 sim · 19.3s → 0.40s
4×transient step 0.2p → 1p
5–6×tuning / pushing sweeps (parallel)
22×Monte-Carlo offset (parallel + fast lib)
  • Root-caused via profiling: sky130's slowness was re-parsing 51 PDK corners every process launch — trimmed to one corner + dropped unused R/C banks (bit-identical).
  • ngspice subprocesses release the GIL → ThreadPool parallelism across DE, NSGA-II, PVT, tuning.
